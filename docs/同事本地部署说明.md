# 同事本地部署说明

这份说明面向第一次接手这个工具、准备在自己电脑上本地运行的同事。

## 1. 你会拿到什么

你会拿到一个 GitHub 仓库地址，例如：

```text
https://github.com/chairlong-create/Cluster-Analysis
```

这个项目是本地单机工具：

- 前端和后端都在同一个 Next.js 项目里
- 数据库存储在你自己电脑上的 SQLite 文件
- 模型配置也只保存在你自己的本地环境里

它不是共享在线系统，所以每个人都需要单独部署、单独配置自己的 `.env`。

## 2. 本机要求

建议版本：

- Node.js 24+
- npm 11+

检查方式：

```bash
node -v
npm -v
```

## 3. 拉代码

```bash
git clone git@github.com:chairlong-create/Cluster-Analysis.git
cd Cluster-Analysis
```

如果你不用 SSH，也可以用 HTTPS clone。

## 4. 安装依赖

```bash
npm install
```

## 5. 配置环境变量

先复制模板：

```bash
cp .env.example .env
```

Windows PowerShell:

```powershell
Copy-Item .env.example .env
```

然后编辑 `.env`。

最重要的是这几项：

- `DATABASE_FILE`
- `OPENAI_COMPATIBLE_API_KEY`
- `OPENAI_COMPATIBLE_BASE_URL`
- `OPENAI_COMPATIBLE_MODEL`

建议：

- `DATABASE_FILE` 每个人都用自己的本地路径
- 不要共用别人的 `.env`
- 不要把自己的 `.env` 提交回仓库

## 6. 推荐的数据库路径

最简单的写法：

```env
DATABASE_FILE="./cluster-analysis-dev.db"
```

这样数据库文件会直接放在项目目录下，但不会被 Git 跟踪。

如果你想放到别的位置，也可以写绝对路径。

## 7. 首次启动前先做检查

```bash
npm run lint
npm test
npm run build
```

这一步的目的不是折腾流程，而是确认你的机器环境、依赖和类型检查都没问题。

## 8. 启动项目

```bash
npm run dev
```

浏览器打开：

```text
http://localhost:3000
```

## 9. 首次使用建议

建议先做小样本验证，不要一上来就跑正式大批量数据。

推荐顺序：

1. 新建一个任务
2. 上传一个小 CSV
3. 先跑一次信号提取
4. 再看聚类建议、类别确认、批量分类是否正常
5. 最后再上正式数据

## 10. Prompt 使用注意事项

可以改：

- 分析目标描述
- 判断口径
- 示例

不要随便改：

- JSON 字段名
- JSON 层级结构

尤其是提取阶段，字段名目前是固定契约。

## 11. 常见问题

### 1）为什么我能打开页面，但提取一直失败？

先去任务日志页看：

- `llm_call_logs`
- `step_run_items`

优先看模型原始 `response_text`，不要只看页面上的失败计数。

### 2）为什么别人能跑，我这边不行？

先排查：

- Node 版本
- `.env` 是否正确
- API Key 是否可用
- 模型 base URL 和 model name 是否匹配

### 3）为什么我想改 Prompt，但一改就解析失败？

通常是因为改坏了结构化 JSON 契约。  
你可以改语义，但不要改返回字段名。
