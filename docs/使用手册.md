# 对话聚类分析工作台使用手册

这份文档面向实际使用者，聚焦“怎么用”，不讲实现细节。

## 1. 使用前准备

建议你先确认三件事：

- 模型配置已经填好
- CSV 文件是 UTF-8 编码
- CSV 的前两列分别是：
  - 第 1 列：`id`
  - 第 2 列：`text`

一次最多可以上传 10 个 CSV。

关于 Prompt 再补一句很重要：

- 你可以改 Prompt 的分析口径和任务目标描述
- 但当前各步骤返回的 JSON 字段名仍是固定契约
- 不要在 Prompt 里擅自改结构化输出的 key 名，否则可能导致解析失败
- 如果提取阶段判断 `has_buy_block_reason=false`，建议仍然返回完整 JSON；最稳妥的做法是把其余字符串字段返回空字符串，`confidence` 返回 `0`

## 2. 什么时候新建一个任务

一个任务，代表一套独立的分析体系。

适合新建任务的情况：

- 你想分析的是一个新的业务问题
- 你不希望和现有类别体系混用
- 你需要单独维护一套类别表

例子：

- 任务 A：分析客户未成交的关键阻碍
- 任务 B：分析客户认可服务的方面
- 任务 C：分析客户持续报名的动因

## 3. 建议的使用顺序

### 第一步：新建任务

在首页创建任务时，至少填写：

- 任务名称
- 任务说明
- 分析目标

分析目标会影响默认 Prompt 的语义。

### 第二步：上传批次

进入任务后，在 `批次推进` 里上传 CSV。

系统默认规则：

- 第一个上传的批次默认是 `建类批次`
- 后续上传的批次默认是 `直接分类批次`

你也可以在 Batch Detail 里手动调整批次用途。

### 第三步：用建类批次搭第一版类别表

建类批次建议按下面顺序走：

1. 信号提取
2. 查看提取样例
3. 生成聚类建议
4. 检查建议是否合理
5. 确认建议写入类别表
6. 再做批量分类

如果提取过程中有失败项，可以用 `失败重试` 单独补跑。

### 第四步：后续批次直接分类

当类别表已经初步成型后，后续批次一般不需要再做建类流程，可以直接：

- 开始批量分类

如果分类结果里 `其他` 变多，说明当前类别表不够覆盖，再去做任务级收敛。

### 第五步：在任务收敛里继续优化类别体系

任务收敛主要有三类动作：

- 行内编辑类别
- 处理全部其他
- 合并近似类别

常见使用方式：

- `其他` 多了：先处理全部其他
- 类别太散了：再合并近似类别
- 某个定义不清晰：直接在 Category Snapshot 表里行内编辑

### 第六步：查看汇总并导出

当任务结果稳定后，可以在任务收敛下方查看：

- 各类别数量
- 各类别占比
- 饼图汇总

然后导出 CSV 明细。

## 4. 批次用途怎么理解

### 建类批次 `seed`

适合：

- 第一次做这个任务
- 当前类别表还不成熟
- 你想用这一批数据帮助系统长出新类别

### 直接分类批次 `classify_only`

适合：

- 当前类别表已经可用
- 你只想快速把新批次按现有类别归进去

## 5. `其他` 怎么处理

`其他` 是系统保留类别。

如果很多记录落在 `其他`，不要一个批次一个批次单独处理，而是用任务级的 `处理全部其他`。

这样做的原因是：

- 可以把所有未被消化的记录放在一起看
- 更容易一次性长出稳定的新类别
- 可以减少同义不同名的重复类别

## 6. 类别表怎么维护

Category Snapshot 现在支持行内管理：

- 编辑类别
- 删除类别
- 新增类别

其中：

- `其他` 不能删除
- `其他` 只能改定义，不能改名称

如果你在新增或编辑时看到红字报错，说明当前行有校验问题，比如：

- 名称为空
- 定义为空
- 名称重复

## 7. 什么时候用“合并近似类别”

适合这些情况：

- 类别数量越来越多
- 有多个类别意思接近，只是名字不同
- 汇总展示里看起来有明显的近义重复项

使用方法：

1. 输入最大合并后类别数
2. 点击 `合并近似类别`
3. 等待建议生成
4. 检查建议是否合理
5. 再决定是否确认应用

注意：

- 这一步会影响类别表和历史结果映射
- 建议先看建议内容，再决定是否确认

## 8. 日志怎么看

日志页主要用来排查：

- 哪一步失败了
- 失败的是哪些条目
- 模型当时返回了什么

重点看三类内容：

- `失败的结构化输出`
- `失败的 LLM 调用`
- `未完整落库的失败 case`

最后一类通常表示：

- 运行过程中有条目提前中断
- run 计数和 item/log 数量对不上
- 但如果现在你已经能在失败项里看到完整 `response_text`，优先先看模型原始输出是不是字段缺失或格式不合法

## 9. 常见建议

- 第一个批次尽量选代表性强的数据做建类
- 不要一上来就上传很多批次直接建类
- 当 `其他` 变多时，优先用任务级处理全部其他
- 当类别开始变散时，再用合并近似类别
- 导出前先看一下任务收敛区的汇总结果是否符合预期

## 10. Prompt 修改注意事项

首页支持直接编辑 Prompt，但目前请按下面规则使用：

- 可以修改：
  - 分析目标描述
  - 判断标准
  - 示例话术
  - 对“其他”或边界情况的处理要求
- 不建议修改：
  - 结构化 JSON 的字段名
  - 返回 JSON 的整体层级结构

以“信号提取”为例，当前代码仍要求模型返回固定 key，例如：

```json
{
  "has_buy_block_reason": true,
  "buy_block_reason": "一句话概括提取出的分析摘要",
  "evidence_quote": "直接引用原文",
  "evidence_explanation": "说明为什么这句原文支持该判断",
  "confidence": 0.85
}
```

虽然这些字段名带有早期版本的历史痕迹，但当前代码会在内部把它们映射成更通用的分析语义。  
因此，现阶段最稳的做法是：

- 改 Prompt 的内容
- 不改 JSON key 名
- 即使判断为“没有未购买原因”，也尽量保持 JSON 结构稳定
